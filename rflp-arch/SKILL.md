# RFLP 全景图

## 概述

生成项目的全景地图——一页纸看清楚项目是什么、有哪些模块、怎么串联。这是 rflp-prd 的前置：设计新功能前先知道项目全貌。

**启动时声明：** "我正在使用 rflp-arch 生成全景图。"

## 边界声明（在做任何事之前先读这段）

**全景图是地图，不是代码审计报告。** 控制在 150 行以内的 markdown，一页纸能看完。

**全景图只包含：**

- 模块总览（5-10 行，每行一句话职责）
- 模块关系（ASCII 图 + 简要数据流）
- 技术栈（主要选型，不列版本号细节）
- 目录结构（关键目录，不列文件）
- RFL 文档索引

**全景图不包含以下内容（这些属于 R/F/L 文档）：**

- 数据库表结构或表清单
- API 端点列表（REST / Dubbo / gRPC / MQ 主题）
- 环境配置详情（多少套环境、数据库地址等）
- 测试覆盖率或测试文件清单
- 具体类名、方法签名或类数量统计
- 依赖版本号明细

如果探索过程中发现了这些信息，**记在心里留给后续 R/F/L 文档使用，不写进全景图**。

## 何时触发

- **新项目**：没有 `docs/` 目录，或有目录但没有 `全景图.md`
- **已有项目**：rflp-prd Phase 1 发现全景图过时或缺失
- **架构变更**：重大重构后，全景图需要更新

## Step 1: 探索项目

**新项目（无代码）：**

1. 跟用户聊：这个项目要做什么？给谁用？核心场景是什么？
2. 从用户描述中提取模块划分的线索

**已有项目：**

1. 读 `README.md`、`CLAUDE.md`、`package.json` / `pyproject.toml` / `pom.xml` 等入口文件
2. 扫目录结构，识别主要模块
3. 读已有的 R/F/L 文档（如果有）
4. 读 `docs/archive/` 历史设计文档
5. 扫关键代码入口（main/index/app），理解模块串联方式

**探索到此为止** — 不深入读具体实现类、不统计类数量、不遍历数据库表、不列举 API 端点。产出是心中一张模块关系草图，不是代码审计报告。

**产出**：心中有一张模块关系草图，再进入 Step 2。

## Step 2: 生成全景图

写到 `docs/全景图.md`，格式：

```markdown
# [项目名] 全景图

> [一句话：这个项目是什么，给谁用]

## 模块总览

| 模块     | 职责         | 对应 R/F/L      |
| -------- | ------------ | --------------- |
| [模块名] | [一句话职责] | R0x / F0x / L0x |

## 模块关系

[ASCII 图或简单描述，展示模块之间的数据流和调用关系]

## 技术栈

| 层面 | 选型                           |
| ---- | ------------------------------ |
| 语言 | [e.g. Python 3.11]             |
| 测试 | [e.g. pytest + tests/ 目录]    |
| 部署 | [e.g. systemd / Docker / 手动] |

## 目录结构

[关键目录说明，不列每个文件]

## RFL 文档索引

| 文档       | 路径                 | 状态          |
| ---------- | -------------------- | ------------- |
| R01 [名称] | docs/R层-需求/R01.md | 已有 / 待创建 |
| F01 [名称] | docs/F层-功能/F01.md | 已有 / 待创建 |
| L01 [名称] | docs/L层-逻辑/L01.md | 已有 / 待创建 |
```

**要求：**

- 模块总览控制在 5-10 行，超过说明模块拆得太细
- 模块关系用 ASCII 图，不用外部工具
- RFL 文档索引列出已有的和需要创建的，标明状态
- 新项目的 RFL 文档全部标"待创建"

**写完后自检（必须逐条过）：**

- [ ] 全文是否在 150 行 markdown 以内？
- [ ] 每个模块职责是否一句话、不超过 15 字？
- [ ] 是否混入了数据库表、API 端点、配置详情、类名统计等详细设计内容？如有，删除
- [ ] 架构关注点（如果有）是否不超过 3 条且每条一句话？
- [ ] 技术栈是否只列选型名称，不列版本号明细？

**如果自检发现越界，先删减再产出，不要「都写了就留着吧」。**

## Step 3: 建 RFL 目录骨架

如果 `docs/` 目录不完整，补全：

```bash
mkdir -p docs/R层-需求 docs/F层-功能 docs/L层-逻辑 docs/plans docs/archive
```

**不写 RFL 文档内容**——只建目录。内容由 rflp-prd 和 rflp-trd 在后续流程中填充。

## Step 4: 提交

```bash
git add docs/全景图.md docs/
git commit -m "docs: 生成项目全景图"
```

## 更新全景图

已有全景图需要更新时：

1. 读现有全景图 + 最近的 RFL 变更
2. 更新模块总览和关系图
3. 更新 RFL 文档索引的状态
4. 提交：`git commit -m "docs: 更新全景图"`

**不重写**——增量更新。用 diff 思维，不用全量思维。

## 红线

**绝不：**

- 在全景图里写详细设计——那是 R/F/L 文档的事
- 给新项目预写 RFL 文档内容——只建目录骨架
- 列出超过 10 个顶级模块——按业务域或部署边界聚合到 10 个以内

**始终：**

- 模块职责用一句话，不超过 15 个字
- RFL 文档索引反映真实状态（已有/待创建）
- ASCII 图展示模块关系，不依赖外部工具

## 记住

- 全景图是地图，不是详细设计——一页纸能看完
- 模块总览 5-10 行，多了就需要合并
- 新项目：只建骨架，不预写内容
- 更新时：增量更新，不全量重写
- 探索时收集的详细信息留给 R/F/L 文档，不塞进全景图
- 这是 rflp-prd 的前置——设计前先看全貌
