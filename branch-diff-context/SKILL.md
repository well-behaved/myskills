# 分支改动快速上下文建立

新 session 开始时，快速读懂当前分支相对基准分支的全部改动，建立后续对话的上下文基础。

---

## 执行流程

### Step 1：确定对比基准

从用户输入中提取目标基准分支：
- 用户明确指定（如"和 master 比"）→ 使用指定分支
- 未指定 → **默认使用 `prod`**

```bash
git branch --show-current          # 当前分支
git branch -a | findstr /i "prod"  # 确认 prod 分支存在
```

### Step 2：获取改动文件列表与统计

```bash
git diff <基准分支>...HEAD --stat
git log <基准分支>..HEAD --oneline
```

### Step 3：读取完整 diff 内容

```bash
git diff <基准分支>...HEAD
```

diff 内容较大时，**按文件逐个读取**，确保每个文件都理解到位：

```bash
git diff <基准分支>...HEAD -- <文件路径>
```

### Step 4：输出结构化改动摘要

按以下格式输出，确保简洁、信息密度高：

```
## 当前分支改动摘要（vs <基准分支>）

**分支**：`<当前分支名>`
**提交数**：N 个 commit

---

### 核心功能（逐条说明每个实质性改动是什么、为什么）

#### 1. <改动标题>
- 涉及文件：`xxx.java`
- 做了什么：<1-2句话说清楚逻辑>

#### 2. <改动标题>
...

---

### 附属改动（参数扩展、配置、DTO字段等）
- `XxxParam`：新增字段 xxx（用途）
- `XxxDTO`：新增字段 xxx（用途）
- 配置文件/gitignore 等微调

---

### 文件清单
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| XxxService.java | 新增方法 | ... |
| XxxServiceImpl.java | 修改 | ... |
```

### Step 5：结尾提示

输出摘要后，补充一句：

> "改动已读完，可以开始提问或提需求。"

---

## 注意事项

- **不评价代码质量**，只描述"做了什么"，评价留给用户主动发起
- **不遗漏文件**，每个变更文件都要在摘要里体现
- diff 较大时优先关注业务逻辑文件（Service/ServiceImpl），配置/DTO 文件可合并描述
- 遇到看不懂的逻辑，直接说"此处逻辑待确认"，不要猜测
