# 测试报错分析与修复规范

---

## 失败类型分类决策树

```
测试失败类型？
│
├─ COMPILATION ERROR（编译失败）
│   ├─ 错误在测试代码 → 修测试代码（import 缺失、类名错误、语法错）
│   └─ 错误在业务代码 → 修业务代码
│
├─ AssertionError（断言失败）
│   ├─ 堆栈指向 src/test/java → 分析断言期望值是否合理
│   │   ├─ 期望值写死但业务已变更 → 修测试（更新期望值）
│   │   └─ 期望值本身就错了 → 修测试
│   └─ 堆栈穿透到 src/main/java → 修业务代码
│
├─ NullPointerException / 运行时异常
│   ├─ 堆栈指向 src/main/java → 修业务代码（null 检查缺失）
│   └─ 堆栈指向 src/test/java → 修测试（测试数据构造有误）
│
├─ BusinessException / 自定义业务异常
│   ├─ 异常信息说"参数缺失/非法" → 修测试（构造正确的入参）
│   └─ 异常信息说"数据不存在/状态错误" → 修测试（使用真实存在的测试数据）
│
├─ ApplicationContextException / Bean 加载失败
│   └─ ⛔ 不自动修复，标记为人工处理（Spring 配置问题）
│
├─ DataAccessException / SQL 错误 / 连接拒绝
│   └─ ⛔ 不自动修复，标记为人工处理（数据库/环境问题）
│
└─ TimeoutException / 端口绑定失败
    └─ ⛔ 不自动修复，标记为人工处理（环境问题）
```

## 堆栈归因规则

判断错误根因属于业务代码还是测试代码：

```
堆栈中最靠近顶部的 cn.sottop.** 包：
├─ cn.sottop.**.service.**  → 业务代码
├─ cn.sottop.**.handler.**  → 业务代码
├─ cn.sottop.**.util.**     → 业务代码（工具类有 bug）
├─ cn.sottop.**.*Test.*     → 测试代码
└─ cn.sottop.**.suite.*     → 测试代码（Suite 里的方法）
```

## 修复执行器任务规范

### 修业务代码时

修复执行器 prompt 必须包含：
1. 失败的完整堆栈（from Surefire XML）
2. 报错行所在的业务方法完整源码
3. 对应测试方法的源码（理解测试意图）
4. 明确指令：**只修最小范围，不做无关重构**

```
硬约束：
- 只修 src/main/java 下的文件
- 不改方法签名（调用方不受影响）
- 不加 @Transactional（项目通过 AOP 统一织入）
- 修复后必须运行 lsp_diagnostics
```

### 修测试代码时

修复执行器 prompt 必须包含：
1. 失败断言的完整信息（expected vs actual）
2. 被测方法当前的源码（确认业务行为）
3. 测试方法当前源码
4. 明确指令：**只更新断言期望值或修正测试入参**

```
硬约束：
- 只修 src/test/java 下的文件
- 不删除测试方法，只修改方法体
- 不改测试方法名
- 不改 @BeforeAll / @AfterAll 基础设施
- 修复后必须运行 lsp_diagnostics
```

## 不能自动修复的情况（必须标记人工处理）

| 情况 | 原因 |
|------|------|
| Spring 上下文启动失败 | Bean 配置/依赖错误，逻辑复杂 |
| 数据库连接失败 / SQL 错误 | 环境或 schema 问题 |
| Nacos/Redis 连接失败 | 中间件环境问题 |
| 端口绑定冲突 | 运行环境问题 |
| 测试断言失败但业务逻辑复杂 | 无法判断期望值是否正确 |
| 同一文件修复超过 2 次仍失败 | 问题超出简单修复范围 |

---

## 修复优先级

多个失败并发时，按以下顺序处理：

1. **编译错误** — 必须先修，否则其他测试无法运行
2. **NPE / 运行时异常（业务代码）** — 明确 bug，高置信度可修
3. **断言失败（测试期望值错）** — 测试代码问题，修测试
4. **断言失败（业务逻辑 bug）** — 修业务代码
5. **其他异常** — 评估是否可修，不确定则人工处理

---

## 回滚规则

修复子任务执行 `lsp_diagnostics` 后：

- **有 error** → 立即回滚该文件（Write 工具写回原内容），记入"回滚列表"
- **无 error** → 记入"成功列表"，进入验证轮

**回滚的文件不再参与后续轮次的修复**，直接标记为人工处理。
