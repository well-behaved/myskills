# RFLP 设计

## 概述

将一个想法变成经过打分验证的 PRD。核心原则：先理解现有标准，再提方案，打分不到 9 分不往下走。

PRD 保存在 `docs/plans/`，**不动正式 R/F 文档**。正式文档在 rflp-finish 验证通过后才合并。

**启动时声明：** "我正在使用 rflp-prd 来分析这个需求。"

## Phase 0: 分流

判断变更类型：新功能、重构、小改动。

**快速通道条件（必须全部满足，缺一不可）：**
- 已在 CLAUDE.md 待办中**明确列出**（不能是"类似的"或"可以推断出的"）
- 改动 ≤ 2 个文件
- 只有一种实现方式，不需要在方案间选择

**不满足任一条件 → 不走快速通道，继续 Phase 1。** 不允许以"改动很小""我有把握"为由绕过。

满足 → 读相关代码和 RFL → 直接实现 → 冒烟测试 → 检查 F 文档是否需要一行更新 → 提交。不产出 PRD，不进入后续流程。

不满足 → 继续 Phase 1。

## Phase 1: 加载上下文

1. 读 CLAUDE.md 待办 + `docs/全景图.md`（如有），确定需求涉及哪些 R/F/L
2. 读 `docs/archive/` 历史设计文档，避免重复设计
3. 浏览所有 R/F 文档标题和首段摘要，列出可能相关的文档，再逐个深读并顺交叉引用找关联模块
4. 特别关注 R 文档中的 AT 规格、F 文档中的 FT 规格——这些是已有的验证标准
5. 探索代码库中相关模块的当前实现

**新项目无 docs/ 目录时：** 标记为"从零创建"，流程不变，Phase 5 输出时标注"需要创建的 RFL 文档路径"。

将已有的 RFL 和 archive 内容作为后续讨论的基础。

## Phase 2: 讨论需求

根据变更类型使用不同的提问框架：

**新功能：**
- 用**场景化语言**提问："用户在什么情况下…"
- 每次只问**一个问题**，优先选择题
- 场景按**用户动机**拆，不按模块拆（"我在桌面编程要出门了"而非"多端切换功能"）
- 场景 ≠ 功能点——功能点拆分是 F 层的事，R 层只管用户动机

**重构：**
- 不问场景化问题，确认三件事：
  1. **接口边界**：哪些调用者看到的行为不能变？
  2. **迁移路径**：数据/状态怎么过渡？
  3. **回滚策略**：出问题了怎么恢复？

**复杂任务：** 主动建议分 session（调研/文档/编码各一个 session）。

## Phase 3: 探索方案

- 提出 **2-3 个方案**（保守/中间/激进），不直接给"最优解"
- 每个方案标出对 RFL 的**影响范围**（主归属模块 + 副影响模块）
- 让用户选方向，用户可能混合方案或提出新方向——调整为一个明确方案再进入 Phase 4

**发现与现有 RFL 矛盾时：** 停下讨论，是更新 RFL 还是调整方案。不强行推进。

## Phase 4: 打分验证

对选定的方案自评：

1. **打分**（1-10）
2. **扣分原因**：具体哪里不够好？
3. **最大风险**：这个方案最可能在哪里翻车？

**打分锚点（新功能）：**
| 分数 | 含义 |
|------|------|
| 9-10 | 能直接交给不了解上下文的人写技术方案，无歧义 |
| 7-8 | 大方向对，但有 1-3 个悬而未决的问题 |
| 5-6 | 核心思路有，但缺关键约束或场景覆盖不全 |
| < 5 | 需求本身还没理清 |

**打分锚点（重构）：**
| 分数 | 含义 |
|------|------|
| 9-10 | 接口边界清晰，迁移路径完整，回滚策略可行 |
| 7-8 | 接口清晰但迁移边界情况未覆盖 |
| 5-6 | 接口边界模糊，部分调用者可能受影响 |
| < 5 | 改动范围不明确 |

**门控规则：**
- ≥ 9 分 → 进入 Phase 5
- < 9 分 → **必须回退 Phase 2 或 Phase 3**，针对扣分原因补充讨论，不允许跳过
  - 不允许以"内部工具""核心流程够了""后面再补"为由跳过回退
  - 不允许将 < 9 分改为 9 分——打分一旦输出，只能通过补充讨论后重新打分，不能直接修改
  - 用户说"够了""可以了"不构成跳过回退的理由
- 3 轮修订仍未达 9 分 → 向用户汇报持续存在的问题，展示：
  ```
  ⚠️ 3 轮修订后仍未达 9 分。
  当前分数：[N]/10
  持续存在的问题：[列出]
  
  选择：
  1. 继续修订（回退 Phase 2/3）
  2. 接受当前方案，在 PRD 中标注已知风险（需你明确输入 'accept-risk'）
  ```
  只有用户输入 `accept-risk` 才能以 < 9 分进入 Phase 5，且必须在 PRD 设计决策中记录。

## Phase 5: 输出 PRD

保存到 `docs/plans/YYYY-MM-DD-<feature>-prd.md`：

```markdown
# [功能名称] PRD

## 设计决策

**目标：** [一句话]
**变更类型：** 功能 / 重构 / 修复
**选择的方案：** [方案名称及核心思路]
**选择理由：** [为什么选这个]
**关键取舍：** [列表]
**打分：** [N]/10
**扣分原因：** [如果有]
**最大风险：** [一句话]

## R 内容

### 场景 X: [场景名]
> [用户动机]

1. [步骤]
2. [步骤]
...

**AT-XX**: [验什么、怎么验、前置条件] [CLI / CLI+人工]

### 场景 Y: ...

## F 内容

> F 层按用户交互入口分组：有 CLI 的按命令分组，有 GUI/Web 的按页面/操作分组，纯 API/SDK 的按模块/类分组。

### 命令参考
| 命令 | 说明 | 示例 |

### 行为规则
- [规则 1]
- [规则 2]

### FT 规格
| FT | 验证 | 方法 |
```

**PRD 内容要求：**
- 设计决策 + R 内容 + F 内容，三部分缺一不可
- AT 规格用自然语言写清"验什么、怎么验、前置条件"，**不写测试代码**
- 1 场景 = 1 AT，AT 必须验证场景**从头到尾全部步骤**，不拆不截尾
- AT 模仿真人操作，**只用 CLI 命令**，不直接调脚本或内部函数
- AT 标注执行方式：`[CLI]` 全自动 / `[CLI + 人工]` 部分人工
- FT 规格从 F 内容直接派生，每条 FT 验证一个具体行为
- FT 围绕单功能测试，用脚本直接准备测试数据（写文件/设状态），不调其他命令，隔离被测模块
- PRD 保存到 `docs/plans/`，不动正式 R/F 文档（避免文档与代码不一致）

## Phase 6: 交接

PRD 确认后，**必须执行以下交接检查清单**：

1. 确认 PRD 文件已保存到 `docs/plans/`
2. 输出完整交接声明（不可简化、不可省略路径）：
   ```
   ✅ PRD 已保存到 `docs/plans/YYYY-MM-DD-xxx-prd.md`。
   打分：[N]/10
   切到 rflp-trd。
   💡 建议新开 session 后执行 /rflp-trd。
   ```
3. 不允许在同一句话里混合交接声明和 trd 内容——声明是一个独立的暂停点

## 何时回退

- Phase 3 发现需求没理清 → 回 Phase 2
- Phase 4 打分 < 9 → 回 Phase 2 或 3
- 任何阶段发现与现有 RFL 矛盾 → 停下讨论
- **不要强行线性推进，质量优先于进度**

## 记住

- 小改动走快速通道，不要用大炮打蚊子
- 先读 RFL 和 archive 再讨论，不要从零设计已有的东西
- 重构用工程语言（接口/迁移/回滚），不强套用户场景
- 2-3 个选项再展开，不直接给最优解
- 打分必须诚实，有锚点对照，追问扣分原因和最大风险
- PRD 必须落文件，不能只存在对话上下文里
- 新增 R 和场景要谨慎，优先升级已有的场景和功能
- 正式 R/F 文档不动——PRD 只存 docs/plans/
